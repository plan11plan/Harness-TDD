# 프로젝트 컨벤션 어댑터 (project-conventions)

> ⚠️ **이 스킬에는 특정 프로젝트·아키텍처 내용이 없다(있으면 안 된다).** 프로젝트 규약은 전부 `slots/conventions`(프로젝트 제공)에 있고, 이 스킬은 그것을 *어디서 읽고 어떻게 적용하는지*만 정의한다. 프로젝트를 바꿔도 이 스킬·에이전트는 그대로이고 슬롯만 교체된다.

## 적용 절차

모든 역할 에이전트는 코드·테스트 작성 전에 `slots/conventions`에서 다음을 읽어 그대로 적용한다:

1. **아키텍처·레이어** — 의존 방향, 책임 경계, 금지 패턴.
2. **테스트 레벨 ↔ 위치 매핑** — 단위/통합/E2E 각 레벨이 코드베이스의 어느 계층·디렉토리에 대응하는지(매니페스트 `levels:`와 함께).
3. **테스트 더블·환경 규약** — 레벨별 격리 수준(단위=의존성 격리 / 통합=실제 인프라 / E2E=엔드투엔드).
4. **네이밍 패턴** — 테스트 이름 규칙(프로젝트 지정 형식).
5. **DB·API 규약** — 삭제·유니크·에러 형식·시간·페이징 등.
6. **객관 룰(있으면)** — 기계 강제 가능한 규칙은 `arch_tool`이 검사. *그런 룰이 없으면 arch_tool 룰 테스트 미생성*, 컨벤션은 Auditor가 주관으로만 검사.

## 원칙

- 슬롯에 *없는* 규약은 가정하지 않는다 — strategist의 pre-flight로 사람에게 올린다.
- 슬롯이 비어 있으면 진행 불가(추론 금지).
- 에이전트는 프로젝트·아키텍처 이름(레이어명·프레임워크·도구)을 *직접 알 필요가 없다* — 항상 이 어댑터를 통해 슬롯에서 읽는다.
