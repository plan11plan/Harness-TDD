# 목표 동작 작성 규약 (target-behavior)

목표 동작 = 루프 에이전트가 *못 보는* 독립 합격 테스트. 에이전트의 자기 테스트가 *틀린 것을 검증하며 초록*인 경우(G1)를 잡는 최종 의미 검증.

## 독립성 규약 (핵심)
- 목표 동작은 **수용 기준(acceptance criteria)에서만** 도출한다. 루프 에이전트가 참조하는 *구현 설계문서*는 보지 않는다 → 둘이 같은 명세 오해를 공유하는 걸 차단.
- 작성 후 **동결**하고 **에이전트 비공개 경로**(예: `_target_behavior/`, 작업 컨텍스트 밖)에 둔다. 평가자만 실행.

## 블랙박스 원칙
- 내부 DTO·구현에 의존하지 않는다. 오직 HTTP 표면(요청→상태코드·응답 바디)·관측 가능한 부수효과만 검증.
- E2E 레벨 완료 후 실행되는 블랙박스 테스트(스택의 E2E 등가물 — 매니페스트 `levels:`가 정의).

## 필수 행동 (빠뜨림 금지 — 입력 템플릿 위험칸과 1:1)
1. **정상 경로** — 대표 성공 케이스.
2. **경계값** — 빈/최대/최소/깊이 제한 등.
3. **권한·소유권** — *남의/없는 리소스 접근 → 거부(권한 없음 코드 — 슬롯 규약)*. (모든 사용자-소유 리소스 엔드포인트마다.)
4. **에러 계약** — 상태코드(없음 404·검증실패 400) + 에러 형식 `{code,message,errors[]}`(또는 프로젝트 규약).
5. **(해당 시) 동시성** — 고위험 연산의 동시 요청 결과.

## 행동 단위 승인 포맷 (사람용)
코드가 아니라 *행동*으로 제시해 승인받는다:

    [목표 동작 검토 — {METHOD} {path}]
    - 입력: <X> → 기대: <상태/결과 Y>     [승인 / 수정]
    - 남의 리소스 접근 → 거부(권한 코드)        [승인 / 수정]
    ...

승인되면 동결. 수정 요청은 *계약 변경*이므로 사람 결정으로 기록.
